# Symmetric-Key Challenge-Response Insight Generator 

This repository contains a comprehensive implementation and simulation of symmetric-key challenge-response authentication protocols, complete with dynamic network routing, active adversarial injection, and comprehensive machine learning models to detect, classify, and predict attacks based on captured network telemetry.

---

## 🔐 Cryptographic Features

The core cryptographic mechanics are implemented using the `cryptography` Python package to guarantee industry-standard security patterns: 

### Core Primitives
- **Shared Secret Key**: 256-bit (32 bytes) symmetric key.
- **Nonce Generation**: 128-bit (16 bytes) cryptographically secure nonces (`secrets.token_bytes`).
- **Encryption Scheme**: AES (Advanced Encryption Standard) operating in **CBC Mode** (Cipher Block Chaining) with **PKCS7** padding.
- **Integrity/HMAC**: Designed with SHA-256 considerations for ensuring message integrity. 
- **Entropy Analysis**: Includes Shannon's entropy computations on network payloads to detect non-randomness caused by attackers.

### Authentication Protocols
1. **One-Way Protocol**:
   - *Bob* generates a challenge nonce $R_B$ and sends it to *Alice*.
   - *Alice* encrypts $R_B$ using the shared key $K$: $E_K(R_B)$ and responds.
   - *Bob* decrypts the response and verifies if the plaintext equals $R_B$.
2. **Two-Way Protocol** (Mutual Authentication):
   - *Alice* initiates the process by sending an initial nonce $R_A$.
   - *Bob* responds with a new nonce $R_B$ and ciphertext $C_1 = E_K(R_A || R_B)$.
   - *Alice* decrypts $C_1$, verifies $R_A$, and sends back $C_2 = E_K(R_B)$.
   - *Bob* decrypts $C_2$ and verifies $R_B$.

## 🌐 Network Architecture

The project features a fully distributed mock-network utilizing low-level TCP sockets and modern Transport Layer Security (TLS):

- **Router/Dispatcher**: A central multi-threaded router that forwards messages between active clients using JSON-encoded payloads over length-prefixed TCP messages. It provides hooks for passive interceptors and active attacker injections.
- **TLS Implementations**: Both *Alice* and *Bob* establish secure TLS (TLSv1.2 - TLSv1.3) contexts for communication over the base TCP layer.
- **Client Entities**:
    - `AliceClient`: Acts as the client handler triggering connections.
    - `BobClient`: Acts as the server/verifier handling responses.
- **Threaded Execution**: Safe execution with Thread locks and concurrent connection handling enables rapid, overlapping session simulations.

## ⚔️ Cyber Attacks & Adversary Simulation

The network explicitly incorporates an Attacker model designed to exploit symmetric key protocols.

1. **Replay Attacks**: The attacker intercepts a legitimate nonce or ciphertext frame and replays it (e.g., repeating an `old_nonce` or sending a previously captured `C1`/`C2` value) to spoof identity without knowing the key.
2. **Random Guess Attacks**: The adversary creates arbitrary or random sequence guesses trying to force a collision or disrupt the session state.
3. **Passive & Active Adversary Model**:
    - *AttackerRecorder*: Passively logs all traffic bypassing the Router. 
    - *AttackerClient*: Actively injects malicious payloads into the router queues during the handshake phases of Alice and Bob.

## 🧠 Machine Learning Models (Attack Classification)

The dataset generated by the `SessionManager` captures deep telemetry representing each authentication interaction (e.g., Response Time, Nonce Entropy, Time Diff, Protocol Type, Sequence Similarity). This data is pipelined into machine learning models located in the `Problem Statements/` folder:

- **Problem 1 (`p1.py`) - Binary Attack Detection:**
  Uses a **Random Forest Classifier** (`n_estimators=200`, `class_weight='balanced'`) to perform binary classification, identifying whether an authentication session is currently under attack (`attack_flag`).
  
- **Problem 2 (`p2.py`) - Multi-Class Attack Classification:**
  Uses a Deep **Random Forest Classifier** to distinguish the *specific type of attack* underway, bucketing network sessions into:
  1. No Attack
  2. Replay Attack
  3. Random Guess Attack
  
- **Problem 3 (`p3.py`) - Authentication Success Prediction:**
  Uses a **Gradient Boosting Classifier** to determine whether an authentication attempt will yield a `success=True` based solely on the structural, temporal, and entropy-related characteristics of the transaction. Evaluates if the attack successfully causes a handshake failure.

## 💻 Simulation & Dataset Generation

To generate telemetry for the ML models, a robust `SessionManager` orchestrates high-speed randomized handshakes.

- **`generate_dataset.py`**: A CLI generator handling multi-threaded batch simulations.
    - *Usage*: `python generate_dataset.py --sessions 5000 --attack-rate 0.2 --replay-prob 0.6`
- **Configuration** (`config/settings.py`): Enables customizing the simulation to toggle Session caps, Key sizes, Attack rates, response delays, and active attackers.
- The output naturally streams out a labeled CSV file `auth_dataset.csv` filled with rich numerical and categorical features of the session traffic.

## 📊 Streamlit Dashboard

A visual user interface is provided via **Streamlit** to rapidly control the environment.

*Run the UI with:* `streamlit run ui/streamlit_app.py`

**Capabilities:**
1. **Live Dataset Generation**: A friendly GUI to tweak parameters (Attack rate, Replay probability, session count) and track live terminal outputs.
2. **Analysis Views**: Displays pandas dataframes summarizing the simulated transactions alongside metrics tracking attack breakdown.
3. **Simulation Runner**: Quick-trigger script for executing pure socket interactions dynamically.
4. **TLS Channel Demo**: Spawns an interactive showcase that establishes an ephemeral TLS channel between Alice and Bob processes and streams both side's respective logs asynchronously to the dashboard.

## 🚀 Getting Started

1. **Install Requirements**:
    Ensure you have `pandas`, `numpy`, `scikit-learn`, `cryptography`, and `streamlit` installed.
2. **Generate the Dataset**:
    Trigger the generator via CLI or Streamlit to establish the `data/results/auth_dataset.csv`.
3. **Run ML Predictors**:
    Execute the Python scripts within `Problem Statements/` to test model accuracies based on your generated telemetry.
4. **Experiment**: Use the variables in `config/settings.py` to test how delays and entropy behavior change depending on varying network conditions.
